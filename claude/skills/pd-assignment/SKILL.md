# pd-assignment

## 前提情報

- 授業: 千葉大学「プログラム設計(pd)」
- 課題ディレクトリ: `~/chiba-u/prog-design/<番号>/`（例: `07` → `~/chiba-u/prog-design/07/`）
- 学生証番号: `24TB4039`
- 前回課題ディレクトリ: `~/chiba-u/prog-design/<番号-1>/`

## ファイル構成

各課題ディレクトリには以下がある:

| ファイル | 説明 |
|---|---|
| `SlidePDxx_2026.pdf` | 講義スライド。課題仕様はここに書いてある |
| `pd0Xa_24tb4000.cpp` | 教員配布スケルトン(a課題) |
| `pd0Xb_24tb4000.cpp` | 教員配布スケルトン(b課題) |
| `pd0Xa_24tb4039.cpp` | **編集対象**(a課題) |
| `pd0Xb_24tb4039.cpp` | **編集対象**(b課題) |

課題によってはスケルトン(`24tb4000`)が配布されず、`24tb4039`のファイルを新規作成するだけの回もある。

## ワークフロー

### 1. PDF確認

`SlidePDxx_2026.pdf` を読んで課題仕様を把握する。PDFは複数ページあるので `pages` パラメータを分けて読む（最大20ページ/回）。課題仕様ページ（`課題pd0Xa`、`課題pd0Xb`のタイトルがある箇所）を重点的に読む。テキスト抽出できるPDFなら `pdftotext -layout` の方が軽量に全体を把握できる。

### 2. 前回課題へのフィードバック確認とスキル更新

**このスキルは毎回自己更新する。** `SlidePDxx_2026.pdf` の冒頭（今回の課題説明が始まる前）には、前回提出した課題への講評・フィードバックが必ず載っている（「前回課題pdXXのフィードバック」「課題レビュー」等の見出し）。ここを読み、以下を行う:

1. **一般化できる指摘かどうかを判断する**
   - 一般化できる例: 「全メンバ変数，メンバ関数にコメント欲しい」「メンバ変数をpublicにしない」「ファイルが開けなかったときのエラーハンドリング」など、書き方・設計の指摘で今後の回にも当てはまるもの → **反映する**
   - 一般化できない例: 「今回の丸め処理の符号が逆」「今回のNクイーンの枝刈りが甘い」など、その回固有の数式・アルゴリズムの間違い → 反映しない
2. **反映済みか確認する**: 末尾の「フィードバック反映ログ」に同じ指摘が既にあれば何もしない
3. **未反映なら、内容に応じて該当セクション（4節の必須ルール／学生らしく書くルール／コメント、5節の発展課題、8節の提出前レビュー、または「よくある落とし穴」）に追記し、「フィードバック反映ログ」に1行追加する**
4. この更新はコード実装より前に行い、以降の節ではすでに更新済みのルールに従う

### 3. スケルトンと自分のファイルを確認

- 教員スケルトン(`24tb4000`)を読んで main() の呼び出しパターンを把握する（配布されていない回はスキップ）
- 自分のファイル(`24tb4039`)を読んで現状を把握する（新規作成の回はスキップ）

### 4. 前回のコードスタイルを参照

前回課題ディレクトリの `pd0Xa_24tb4039.cpp` / `pd0Xb_24tb4039.cpp` を読む。以下を合わせる:
- コメントスタイル（日本語、短め）
- インデント・スペーシング
- `std::cout` の使い方

### 5. 実装

**必須ルール:**
- メンバ変数名は `_` プレフィックス（例: `_rows`, `_year`）
- コンストラクタと Set() で処理を重複させない → コンストラクタは対応する Set() を呼ぶだけにする（`init()`のような別関数だと呼び忘れるリスクがあるため、コンストラクタでの初期化が推奨されている）
- 動的メモリ(`new`)を使ったらデストラクタで `delete[]`
- 代入演算子(`operator=`)もコピーコンストラクタと同様にシャローコピーを避ける（既存メモリを解放してから新規確保し、要素を1つずつコピーする）
- メンバ変数は原則 private にする（外部から直接触らせない。カプセル化）
- コンストラクタ以降で使わない値まで不要にメンバ変数化しない（例: コンストラクタでしか使わないファイル名などはローカル変数で十分）
- 仮引数名はメンバ変数名（`_`なしの素の名前）と一致させない → 同名にすると意図せずシャドーイングし、想定外の動作になる
- ファイルを開く処理(`fopen`, `ifstream`)は失敗時のチェックを入れる（開けなければエラーメッセージを表示して終了する）
- 教員が「汎用テンプレートを積極的に活用してほしい」と明言しているため、汎用化できる処理は可能な範囲で `template` を使う

**コメント（必須）:**
- **全てのメンバ変数・メンバ関数にコメントを付ける**（関数だけでなくメンバ変数側も。コメント漏れは毎回指摘されている）
- 純粋仮想関数のオーバーライドなど「オーバーライドだと分かる」関数でも、「オーバーライド」とだけ書かず中身の説明を書く
- ファイル冒頭の「説明:」に工夫した点を箇条書き
- 調べて使った関数があれば「調べた関数:」に記載

**学生らしく書くルール（重要）:**
- 関数名は英語でも短くシンプルに（例: `eraToYear` ○、`eraToWestern` × 、`parseString` × → `setDate` ○）
- ローマ字化した日本語の関数名は避ける（例: `hantei()` ×）。英語として意味が通る名前にする
- 配列・コレクションを表す変数名は複数形にする（例: `city` → `cities`）
- 変数名はキャメルケースにする。スネークケース(`saved_number`)より`savedNumber`が推奨されている
- `<iomanip>` の `setw/fixed/setprecision` より `printf("%6.2f", ...)` の方が自然
- `struct tm t = {}` より `memset(&t, 0, sizeof(t))` の方が学生らしい
- 過度な抽象化・private helper の細分化を避ける
- AI臭のある命名（`compute`, `process`, `handle`, `manage`, `western`）を避ける
- **講義で出ていない標準ライブラリ関数を避ける**。例えば2要素の入れ替えは `std::swap`（+`<algorithm>`）より一時変数を使った手動swapの方が学生らしい。`std::sort` 等も同様。使う場合は「調べた関数:」に書けるものだけにする
- 可能な範囲で、計算結果は不要な中間オブジェクト変数を作らず直接 `return` する方が簡潔（教員が「簡潔な実装」として紹介している）。ただし可読性を落としてまで詰め込まない

### 6. 発展課題の追加（高得点化）

課題本体を実装したら、必ず発展要素を加えて高得点を狙う。

1. **PDFに発展要素の記載があるか確認する**
   - 「発展」「発展例」「その他発展」「発展として」等のタイトル・箇条書きを探す（課題ページだけでなく前後の講評スライドにもあることが多い）
   - **ある場合はそれを実装する**（PDFの指示が最優先）
2. **PDFに発展要素の記載がない場合は、学生が思いつく範囲で発展を考える**
   - 演算子課題 → 複合代入(`+=,-=,*=,/=`)、比較(`==,!=`)、単項マイナス、ゼロ除算・サイズ不一致のエラーチェック、求めた解の検算
   - クラス課題 → 例外/エラーメッセージ、`ToString()`/`Norm()` 等の便利関数、表示の改良（`0.00`を出さない等）、標準入力対応
   - 描画課題 → 図形（長方形・円）の描画、塗りつぶし、2画像並記
3. **やりすぎない**。テーマを1本に絞り、AIっぽくならない範囲で2〜3個。既存の演算子を再利用して実装する（例: `+=` は `*this = *this + x`）と学生らしく重複も減る
4. **発展分も main に必ず動作検証コードを書く**（PDFが「動作検証コードを実装すること」と要求している場合が多い）。発展だけでなく、標準仕様側に複数パターン・オプションがある場合（例: 回転角度0〜3の全パターン）はそれらすべてを検証コードでカバーする

### 7. コンパイル確認

```bash
g++ pd0Xa_24tb4039.cpp -Wall -Wextra -std=c++17 -pedantic -o /tmp/pd0Xa
/tmp/pd0Xa
```

バイナリは必ず `/tmp/` に出力してカレントディレクトリを汚さない。警告ゼロを確認する。

### 8. 提出前レビュー（AI臭チェック）

コミット前に、**完成した各ファイルをもう一度最初から最後まで Read する**。以下のAI臭がないか確認し、あれば修正して再コンパイルする:

- 過度な抽象化・不要な private helper の細分化
- 不自然/大げさな命名（`compute`, `process`, `handle`, `manage` 等）
- 冗長・説明過多なコメント、英語コメントの混在
- 講義で習っていない標準ライブラリ関数の使用
- 前回課題とスタイル（インデント・コメント量・命名）がズレていないか
- **課題PDFの仕様と完全に一致しているか**（「仕様との不整合」は繰り返し指摘される減点ポイント）
- 全メンバ変数・メンバ関数にコメントが付いているか

### 9. コミット・プッシュ

```bash
git add pd0Xa_24tb4039.cpp pd0Xb_24tb4039.cpp
git commit -m "finish 0Xa" # または "finish 0Xb"
git push
```

### 10. ユーザーへの報告

最終応答で、**どこを発展要素として追加したかを明示する**（PDF由来か自分で考えたかも区別する）。検算結果や期待値との一致も併せて報告する。スキルを更新した場合は何を反映したかも一言添える。

## よくある落とし穴

- 過去の日付の日数差で `+ 0.5` rounding → 負の値は `- 0.5` が正しい
- `localtime()` の戻り値ポインタは静的領域 → 使う前に struct にコピーする: `struct tm t = *localtime(&now);`
- コピーコンストラクタで `_data` をそのままコピーするとシャローコピーになる → `new` して要素を1つずつコピー
