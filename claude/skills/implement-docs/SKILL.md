# implement-docs

`docs/` にある設計書（=あなたの認証・手入力を待たずに実装完了まで行けるよう書かれた自己完結ドキュメント）から、
プロジェクトを**無人で**実装し、別系統・最上位モデルのレビューを合格まで回して修正し、`main` マージの一歩手前で止める。

このスキルの価値は「あなたが寝ている間に、実装からレビュー往復・修正・push まで完結し、朝は main マージを承認するだけ」
にすること。だから**前提が崩れていないか（設計書が本当に自律実装可能か）を自分で保証してから走る**のが核になる。

関連スキルとの棲み分け:
- `plan-project` / `design-strategy` … 設計書（docs）を**書く**側。上流。
- 本スキル `implement-docs` … その docs から**実装しレビューする**側。下流。
- `plan-handoff` … 別セッションへ委譲する用途。本スキルは同セッションで実装まで走る点が違う。

## 起動と引数

```
/implement-docs [path-or-glob]
```

- 引数省略時の対象は `docs/*.md`（**トップレベルのみ**。`docs/reviews/` 配下のレビュー生成物を spec と誤読しないため）。
- 引数の docs 群を「実装すべき唯一の正典」として扱う。会話の記憶がゼロでもこれだけで実装できる、という前提で読む。

## Phase 0 — 設計書レディネス審査（実装より前・必須ゲート）

**なぜ最初にやるか:** 設計書が未完成のまま無人実行に入ると、夜通し走って途中で認証待ちに詰まる／曖昧な箇所を推測で
埋めて破綻する、という最悪パターンになる。無人実行の前提そのものをここで検証する。合格しない限り実装に入らない。

対象 docs を全読し、次を判定する（`plan-project` / CLI完結原則に準拠）:

- [ ] **自己完結**: パス・コマンド・決定事項が明記され、この会話の記憶ゼロでも単独で追える。
- [ ] **認証待ちゼロ**: 途中でユーザーの認証・対話ログイン・手入力が要る手順が無い。要る場合は、その CLI が
      **既に認証済み**であることを実測で確認する（`vercel whoami` / `neonctl me` / `supabase projects list` など。
      **verify, don't assume** ―「たぶん使える」で進めない）。
- [ ] **DoD/受け入れ基準**: 完成の定義があり、後段レビュアの合否判定の的になる。
- [ ] **未決なし**: `要確認` / 未解決 TODO / ビルドを左右する曖昧な選択が残っていない。
- [ ] **依存の実在**: 参照するサービス・CLI・リポジトリが実在し使える状態。

**実行主体:** 主にメインループがチェックリスト＋CLI 実測で判定する。完成度の主観判定が際どいときだけ、後段と同じ
最上位モデルの spec 監査サブエージェント（readonly）に上げてよい。

**不合格時の挙動:** 実装に入らず**停止**する。埋まっていない穴を具体的に列挙し（どの doc の何が足りないか）、
「docs を直す / それでも進める / 中止」をあなたに選ばせる。**推測で埋めて突っ込まない。**

## Phase 1 — 実装ループ（合格まで反復・上限3）

1 ラウンド = **実装 → 自己検証 → レビュー → 修正**。以下を最大 **3 ラウンド**回す。

1. **実装** — docs 通りに実装する。途中で認証／不可逆ゲート（push・本番DB適用など Phase 0 で見落とした手動要件）に
   当たったら、設計書前提が破れたとみなし停止・報告する（Phase 0 に差し戻す）。
2. **自己検証** — typecheck / lint / test / 実際に動かす、で動作を実測する（`verify` 文化）。**ここを通さずにレビューへ
   渡さない。** 壊れたものをレビューさせるのは往復の無駄。
3. **レビュー** — 下記「レビューサブエージェント仕様」に従い 1 体起動。指摘を `docs/reviews/round-N.md` に日本語で
   書かせ、構造化 verdict を受け取る。
4. **判定** — verdict の blocking 指摘が無ければ**合格** → Phase 2 へ。あれば 5 へ。
5. **修正** — blocking を潰す。round++ で 2 に戻る。

**上限到達（3 ラウンド消化してなお blocking 残存）:** **黙って握らない。** 残指摘を `docs/reviews/round-N.md` と完了
サマリに明記し、Phase 2 の push まではやったうえで「未解決あり」として停止・報告する。

### レビューサブエージェント仕様

Agent ツールでサブエージェントを 1 体、**readonly**（コードを編集させない）で起動する。修正は本体が行う
＝レビューと修正の分離。これはあなたの既存方針（review は readonly・検証必須・高シグナル束ね）の踏襲。

- **モデル**: 別系統の**最上位を 1 体**。実装は本体（Opus 系）なので、レビューは**別系統**（既定 Fable 5＝`model: "fable"`）を
  選び、モデル多様性で盲点を突く。
  - **自動降格ポリシー**: 優先順は `fable → opus → sonnet`。指定モデルが無効（例: Fable 5 引退）になったら、その時点の
    priority で**次に有効な最上位へ自動で降格**する。つまり Fable 5 が消えても手を入れずに最上位が回り続ける。
- **工数**: reasoning effort は**最高**。効率化のために手を抜かせない。指定できる経路（専用 agent 定義の frontmatter や
  Workflow の `effort` 等）があればそれで最高に固定し、無ければレビュアのプロンプトで「網羅的・敵対的に、
  最大深度で検証せよ」と明示する。
- **姿勢**: 敵対的レビュー。**高シグナルに束ねる**（nitpick・好みは除外し、壊れる/仕様を満たさない/セキュリティ/データ
  破壊につながる指摘に絞る）。**主張は必ず自分で検証**してから書く(該当コードを読む・再現条件を示す)。
- **入力**: 対象 docs（正典）＋ 実装差分＋自己検証の結果。「docs の DoD を満たしているか」を軸に見る。
- **出力（2 つ）**:
  1. `docs/reviews/round-N.md` に日本語のレビュー本文（あなたが朝読む用。従来の docs/*.md レビューと同じ形式）。
  2. 本体が判定に使う**構造化 verdict**。最低限この形:

```
verdict: pass | changes-needed
blocking:            # これがある限りループ継続。無ければ合格
  - file: path:line
    problem: 何がどう壊れる/仕様を満たさないか
    repro: 再現条件・根拠
non_blocking:        # 参考。ループは止めない
  - ...
```

## Phase 2 — 無人で走る範囲 と 唯一の承認ゲート

**自律（あなたの確認不要）でここまで走る:**
実装 → 自己検証 → レビュー → 修正 → **ブランチに commit** → **push** → **PR 作成** → **preview/dev/staging デプロイ** →
**dev/staging の DB 操作（マイグレーション含む）**。

**唯一の確認ゲート = `main` へのマージ。**
ここだけ停止し、朝の承認を待つ。PR は作って push まで済ませ、「**マージ可の状態**」で止める。本番デプロイ・本番DB適用は
実質この main マージに紐づくので、同じゲートで守られる。

### ガード（あなたの既存ルールと両立・新指示を上書きしない）

- **main 直コミット禁止**。必ず先にブランチを切る（`push.autoSetupRemote` 前提）。
- **コミットメッセージに Claude / Co-authored-by を含めない**（あなたの Contribution 規約）。
- **本番 DB は値を触る前にバックアップ**（Neon/Supabase 方針）。dev/staging は自由。
  破壊的操作（drop/truncate 等）を**本番に対して**行う場合のみ確認する。
- **削除は `rm` 禁止 → `trash -v`**。rm が必要と判断したら理由を説明して提案する。

---

## 完了サマリ（朝レビュー用・必ず残す）

停止時に、あなたが最短で承認判断できるよう次を出す:

- **実装概要**: 何を作ったか、対象 docs の DoD を満たしたか。
- **ブランチ / PR**: ブランチ名・PR URL（作成済みなら）。
- **レビュー履歴**: `docs/reviews/round-*.md` へのポインタと各 round の要点。
- **残件**: 上限到達で残った blocking があれば具体的に（無ければ「なし」と明記）。
- **次の一手**: あなたがやるのは「PR を確認して main へマージ」だけ、という状態にする。

| 禁止 | main 直コミット / コミットに Claude 名 / 本番DBバックアップ前操作 / `rm` |
